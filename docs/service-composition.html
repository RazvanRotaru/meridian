Design for change,
not just for running.

The Service composition tab reframes the first view of a codebase. Instead of "what calls what at runtime," it answers the architect's question: are these the right units, and are they coupled and cohesive enough that I can change one without breaking the others?

The reframe

Two views, two questions

A call graph tells you how execution moves. It's the right lens when you're tracing a bug or following a request. But it doesn't tell you whether the code is shaped well — whether a module does one job, whether a class is a hub everything leans on, whether a dependency points the wrong way. Those are the questions you ask when you're about to build or refactor something. So the first tab now answers those instead.

The load-bearing idea

The person who coined SOLID — Robert C. Martin — also defined the metrics for reasoning about how components fit together. We compute exactly those, from the call/instantiate/inherit edges the extractor already produces. No new data, no dependency-graph dump — just an honest reading of the structure you already have.

The unit

Each box is a scorecard, not a folder

A unit is a class, an interface, an object literal, or a whole module (its loose functions). Every unit becomes a card whose numbers are derived from the graph — and whose left rail is coloured by health, so a problem area is visible before you read a single digit.

The five numbers

What each metric means — and which principle it guards

Ce · CaCoupling, out & inSRP · DIP

Distinct units this one depends on (Ce, efferent) and that depend on it (Ca, afferent). Fan-out is responsibility; fan-in is risk.

IInstabilityDIP

How exposed a unit is to change from below.

I = Ce / (Ce + Ca) — 0 stable, 1 unstable

AAbstractnessOCP · DIP

Share of a unit that is abstract (interfaces, abstract members). Stable things should be abstract so you extend rather than edit them.

DDistance from the main sequencethe score

How far the unit is from the ideal balance of abstraction and stability.

D = | A + I − 1 | — 0 ideal, 1 worst; this colours the rail

CohCohesion (LCOM4)SRP

Do a unit's own methods work together? Measured as connected clusters of internal calls. Many clusters → it's really several units in a trench-coat.

⌖Smellsread-out

HUB coupled both ways · PAIN concrete & depended-upon · UNUSED abstract & friendless · SPLIT low cohesion. The chips are just thresholds crossed on the numbers above.

The map behind the colour

The main sequence

Plot every unit by abstractness (A) against instability (I). The diagonal is the main sequence — the healthy balance where stable code is abstract and volatile code is concrete. The two corners off it are the traps. A unit's distance from that line is the health colour on its rail.

Zone of pain — concrete and heavily depended-upon. Rigid: everyone needs it, nobody can safely change it. On shopfront, response.ts and BaseRepository live here.
Zone of uselessness — abstract but nothing depends on it. Speculative generality; StoredInventory is an interface no one uses.
On the sequence — abstraction matched to stability. The goal for anything you build.
Most real TypeScript clusters along the bottom edge (A≈0 — few abstractions), so health there is driven almost entirely by instability. That flatness is itself a finding.

Where things live

Packages, and the wires that cross them

Units are grouped into a titled frame per package (folder) — the view reads as "what's in each package," and a frame carries an N⚠ badge when it holds smelly units. The wires are colour-coded by whether they stay home:

Internal — coupling within a package. Expected cohesion; it recedes into the background.
Cross-boundary — a dependency that leaves its package. This is the packaging signal: a unit more wired across a line than within it may be in the wrong place.
Inheritance — extends / implements, called out separately because it's a different kind of relationship.

Why this matters

When most of a system's dependencies cross package lines, the folders aren't really encapsulating anything — the packaging is decorative. That's a Common-Closure signal you can act on: move the unit, or redraw the boundary.

Where you start

Rooted where you are, not the whole system at once

A whole-codebase dump is a hairball. So the tab opens rooted at one file — by default the app entry — showing the units that file contains plus their one-hop neighbours as faded boundary cards. Click a boundary card to travel into it; press ⌘P to root anywhere; the breadcrumb's "Whole system" zooms back out. You explore the composition the way you read code: from a point of interest, outward.

Keeping it honest

What this is — and isn't

The metrics are a lens, not a verdict. The tab is deliberate about what it can and can't claim:

  • ▹
    Derived from real edges, always in sync. Coupling comes from calls, instantiates, extends, implements the extractor already emits — regenerate the graph and the scores follow the code.
  • ▹
    Not a dependency graph. It's a scored composition map, not a wire-dump. import edges are a separate optional layer (a fast-follow), off by default so the signal stays clean.
  • ▹
    Cohesion is call-based, so it reads pessimistic in TypeScript. Methods often reach through this.repo.x() rather than calling each other, so SPLIT fires broadly. Treat it as a prompt to look, not proof.
  • ▹
    The thresholds are opinions. HUB/PAIN/UNUSED/SPLIT fire on tunable constants. They're conversation-starters for "how should we build this," which was the whole point.
