# Request telemetry POC

Status: Implemented POC
Scope: TypeScript, local `meridian view`, deterministic demo data
Fixture: `examples/orders-service`

## Outcome

Meridian should connect one observed request to the static code map so a reader can answer:

1. Which code ran, in what causal order, and for how long?
2. Which input or state caused each observed branch?
3. Which values were produced and consumed by later steps?
4. Which work failed, was skipped, or continued asynchronously?
5. Where in source can the reader inspect or add the relevant telemetry?

The POC has two coordinated inspection surfaces. The Map, Service, UI, and extracted graph views
project the selected request onto the static graph; the Request trace view keeps the AppMap-style
waterfall for precise timing and event inspection. Both read the same selected trace and use real
elapsed time, not the pseudo-time generated by the static Timeline view.

## Current state and gaps

Meridian started with a stable function join key (`GraphNode.id`), source locations, extracted logic
flows, an environment gate, and an aggregate telemetry overlay. The POC closes three original gaps:

- `Overlay` contains only per-node aggregate metrics. It has no request, span, event, or causal data.
- The current Logic Timeline visualizes possible static execution with invented ticks. It is not an
  observed request.
- Extracted `FlowStep` and `FlowPath` values now carry portable source anchors and semantic path IDs.
  They enable the fixture correlation while the production manifest still needs generated site IDs.

Request traces are therefore a separate contract and endpoint. Aggregate metrics remain unchanged.

## Product requirements

### Request selection

- Request telemetry is an explicit presentation mode. Outside **Telemetry** mode the source picker,
  request controls, runtime card badges, and request paint stay hidden; leaving the mode preserves
  the loaded bundle so returning does not require another fetch.
- Local `meridian view` advertises an in-app source catalog even when it was launched without
  `--overlay`. The POC catalog always includes **Synthetic demo**; a configured saved overlay is an
  additional opaque choice. Browser input never contains or selects a server filesystem path.
- Source selection starts **Off** unless the CLI explicitly configured a source. Choosing a source
  only reveals its bounded environment list; it never fetches data. Loading remains an explicit
  reader action, and the one-click demo shortcut names exactly what it will load.
- Telemetry remains behind the mandatory environment selection; production is never implicit.
- The UI lists bounded request recordings for the selected environment and never downloads an
  unbounded trace history.
- Each request summary exposes its route/name, status, duration, start time, completeness, trace
  ID, and source. The current POC ships at least twelve deterministic recordings that are always labeled
  **synthetic demo**, including when someone explicitly selects another environment.
- A trace whose graph reference does not match the loaded artifact is shown as mismatched. Meridian
  may use fallback mapping for inspection, but must not claim an exact join.

### Timeline

- Use one horizontal millisecond axis and one row per span. Bar width represents observed duration.
- Indentation and parent connectors expose the call tree. Parent IDs express nested causality;
  span links express detached, queued, or cross-trace causality.
- Clicking a row selects the corresponding `node.id`; double-clicking a mapped callable opens its
  static Logic flow. Unmapped work stays visible and is labeled rather than guessed.
- Branch, data, loop, async-handoff, and exception events are keyboard-selectable pins on the span
  that carried them.
- Selecting an event shows its source anchor, outcome, and allowlisted attributes. Alternative static
  paths may be dimmed, but must not be rendered as if they executed.
- The view exposes incomplete/dropped data explicitly. Ordering uses parent/link causality first and
  timestamps second, so clock skew cannot silently reverse distributed work.

### Graph map overlay

- The selected request is a paint-only layer over the existing Map, Service, UI, and extracted graph
  geometry. Switching requests must never relayout cards or change normal graph selection.
- A span paints only after an exact `nodeId` join against the matching artifact. Collapsed files,
  packages, service frames, and service domains may show a roll-up of exact descendant evidence.
  They must identify that evidence as contained work rather than claim a synthetic container span.
- Cards expose first-seen order, occurrence count, observed inclusive duration, event counts, and
  status. Multiple spans on one callable remain separate occurrences in the model and aggregate only
  for display.
- A wire lights only when a causal parent/child span pair maps to one unique static execution edge.
  For the POC, execution edges are `calls` and `instantiates`; adjacency must never light an import,
  reference, ownership, or render edge.
- Bundles, ribbons, cycles, buses, and spools retain `underlyingEdgeIds`, so the runtime layer follows
  proven leaf-edge evidence through the existing presentation pipeline instead of rematching visual
  endpoints.
- Unpainted cards are labeled **not observed**, not **did not execute**. A complete capture means the
  producer reported no dropped data; it does not prove that every possible node or branch site was
  instrumented. Definitive negative evidence requires a versioned instrumentation-coverage plan.
- Graph-reference mismatch disables graph painting as a whole while leaving the request timeline
  inspectable. Missing or ambiguous runtime work is counted and remains visible in timeline/runtime
  diagnostics rather than being guessed onto the graph.
- Runtime highlighting composes with the ordinary selection ring and wire inspector. Selecting a
  dimmed card may make it more readable for inspection, but cannot turn it into observed evidence.

### Request logic flow overlay

- The request split paints the captured runtime-occurrence causal graph as the telemetry path.
  Parent/child calls, span-local event order, links, and explicit async handoffs remain distinct;
  overlapping siblings are never joined merely because their timestamps are adjacent. Those edges
  are visually stronger and retain a positive evidence record keyed by trace and occurrence IDs.
- Expanding a runtime occurrence grafts its extracted static Logic body into the card. Extracted
  `FlowStep` values carry portable source anchors and `FlowPath` values carry semantic path IDs, so a
  branch/loop event can light the correlated inner path for that span occurrence. Straight-line edges in
  a complete, normally completed span and proven sequential child calls carry their own evidence basis.
- Inner wires without a successful source/path or span-order proof remain subdued **code context**. An
  untaken path and an uninstrumented path deliberately share that neutral treatment; neither is
  labeled “did not execute.”
- Missing evidence is always unknown. A reader must never infer that a dim static branch was skipped
  merely because the current producer did not correlate it.
- A future caller `callSiteId` on child spans is still required to disambiguate repeated calls to one
  target in production. Generated static `siteId` values, project-relative paths, and columns are
  also required to remove the POC's basename/line fallback. The current POC uses ordered exact-target
  children only for straight-line call sequences and never uses that weaker join to select a branch.

### Data safety

- Raw arguments, request bodies, headers, cookies, authorization data, SQL binds, and arbitrary
  return values are off by default.
- Default captures are type/shape, counts, booleans, enums, bounded monetary values, and opaque
  correlation tokens. Value capture requires a per-site allowlist, redaction policy, maximum length,
  and classification.
- Captured values are bounded event fields, never labels or span names, to avoid accidental
  high-cardinality indexes.
- The POC schema caps a recording at 2,000 spans/events and bounds text, attributes, and arrays.
  Production ingestion must also enforce a 256 KiB serialized limit. Preserve the trace skeleton
  before detail under backpressure and report every dropped span/event/value count.
- Initial performance budget: under 1% on unsampled paths and no more than 2 ms or 5% p95 latency
  overhead on sampled requests. Capture work is guarded by the SDK recording/sampling decision and
  export is asynchronous.

## Runtime contract

The wire model follows OpenTelemetry's trace model: a request is a trace, timed operations are spans,
and parent IDs/links carry causality. Current stable code coordinates are `code.function.name`,
`code.file.path`, and `code.line.number`; Meridian's exact custom join remains
`meridian.node.id`. See the [OpenTelemetry trace API](https://opentelemetry.io/docs/specs/otel/trace/api/),
[HTTP span conventions](https://opentelemetry.io/docs/specs/semconv/http/http-spans/), and
[code attribute registry](https://opentelemetry.io/docs/specs/semconv/registry/attributes/code/).

OpenTelemetry is moving new event emission toward trace-correlated log events while retaining trace
views and compatibility for existing span events. Meridian normalizes both inputs into the same
`TimelineEvent` model instead of binding the product to one SDK emission API. See
[the Span Event API transition](https://opentelemetry.io/blog/2026/deprecating-span-events/).

```ts
type AttributeValue =
  | string
  | number
  | boolean
  | string[]
  | number[]
  | boolean[];

interface TraceBundle {
  traceVersion: "1.0.0";
  source: "mock" | "tempo";
  env: string;
  generatedAt: string;
  graphRef: {
    schemaVersion: string;
    generatedAt: string;
    nodeCount: number;
    commit?: string;
  };
  traces: RequestTrace[];
}

interface RequestTrace {
  traceId: string;
  name: string;
  rootSpanId: string;
  startedAtUnixNano: string; // decimal strings preserve uint64 precision
  endedAtUnixNano: string;
  status: "unset" | "ok" | "error";
  attributes: Record<string, AttributeValue>;
  spans: TimelineSpan[];
  completeness: {
    complete: boolean;
    droppedSpans: number;
    droppedEvents: number;
    droppedValues: number;
  };
}

interface TimelineSpan {
  spanId: string;
  parentSpanId?: string;
  nodeId?: string;       // exact GraphNode.id; never a parallel function identifier
  name: string;
  kind: "internal" | "server" | "client" | "producer" | "consumer";
  startedAtUnixNano: string;
  endedAtUnixNano: string;
  status: "unset" | "ok" | "error";
  attributes: Record<string, AttributeValue>;
  links?: Array<{
    traceId: string;
    spanId: string;
    relation: "async" | "message" | "detached";
    attributes: Record<string, AttributeValue>;
  }>;
  events: TimelineEvent[];
}

type TimelineEvent =
  | {
      type: "branch.taken";
      eventId: string;
      timeUnixNano: string;
      attributes: Record<string, AttributeValue>;
      siteId: string;
      pathId: string;
      condition: string;
      outcome: string | number | boolean;
      source: { file: string; line: number; col?: number };
      valueName?: string;
      value?: AttributeValue;
    }
  | {
      type: "data.observe";
      eventId: string;
      timeUnixNano: string;
      attributes: Record<string, AttributeValue>;
      name: string;
      valueId: string;
      value: AttributeValue;
      siteId?: string;
      source?: { file: string; line: number; col?: number };
      derivedFrom?: string[];
    }
  | {
      type: "loop.summary";
      eventId: string;
      timeUnixNano: string;
      attributes: Record<string, AttributeValue>;
      siteId: string;
      label: string;
      iterations: number;
      emittedIterations: number;
      truncated: boolean;
      source: { file: string; line: number; col?: number };
    }
  | {
      type: "async.handoff";
      eventId: string;
      timeUnixNano: string;
      attributes: Record<string, AttributeValue>;
      siteId: string;
      mode: "awaited" | "detached" | "callback";
      source: { file: string; line: number; col?: number };
      targetSpanId?: string;
    }
  | {
      type: "exception";
      eventId: string;
      timeUnixNano: string;
      attributes: Record<string, AttributeValue>;
      exceptionType: string;
      message?: string;

      siteId?: string;
      source?: { file: string; line: number; col?: number };
    };
```

Spans encode call/start and return/end, so the contract does not duplicate them as separate events.
The renderer can derive AppMap-style enter/exit markers. AppMap remains the UX reference because its
recordings combine code structure, calls, timing, parameters/returns, I/O, and exceptions; Meridian
keeps the static graph and runtime recording separate so the same graph can accept OTel data. See
[AppMap recording concepts](https://appmap.io/docs/get-started-with-appmap/making-appmap-data.html).

## Joining runtime to the graph

Join in this order and report the mapping quality:

1. `meridian.node.id` -> exact `GraphNode.id`.
2. `code.function.name` -> the node's qualified runtime name.
3. `code.file.path` plus `code.line.number` -> a containing node location.
4. A bounded span-name hint -> fuzzy, inspection-only mapping.

Branch and call-site events require two additive static fields before production instrumentation:

- `siteId` plus source location on every call, branch, loop, callback, and exit step.
- `pathId` on each branch path, including synthetic `else` / `no match` paths.

These IDs are deterministic within one graph artifact, not promised across commits. The POC compares
the trace's schema version, generation timestamp, node count, and optional commit with the loaded
graph. Before production, both the runtime and instrumentation plan must carry an exact artifact
digest. A mismatch is explicit; Meridian never maps a branch by renderer array index.

## What can be automatic

| Capability | POC/production approach | Automation |
| --- | --- | --- |
| HTTP server/client, fetch, database, messaging | OpenTelemetry framework instrumentation | High |
| Request context, parent/child spans, errors | OTel SDK + context propagation | High |
| Selected internal functions | generated build/loader transform | High |
| Exact `if` / `switch` / `try` outcome | transform-generated event at `siteId` / `pathId` | High after site IDs |
| Loop count | generated counter plus capped summary | High after site IDs |
| Selected arguments/returns | explicit allowlisted capture plan | Medium |
| Same-object flow within one process | weak-map opaque value tokens | Medium |
| Full variable/field lineage | compiler-level taint tracking | Out of POC |
| Reflection/native/dynamic calls | preserve as unmapped runtime work | Incomplete by nature |

## Graph-generated annotations without handwritten boilerplate

Yes, but the safe implementation is a generated instrumentation plan, not free-form string edits
from React:

1. The reader selects a callable or flow step and chooses **Trace**, **Record branch**, or
   **Capture value**.
2. Meridian writes `.meridian/instrumentation.json`, keyed by artifact identity (an exact digest in
   production), `nodeId`, `siteId`, and capture policy.
3. A TypeScript loader/build transform injects the SDK calls into generated output. Source can stay
   unchanged.
4. If source annotations are desired, export an idempotent AST codemod patch and preview it before
   applying. Never patch by line-oriented string insertion.

Function probes are feasible with today's node IDs and locations. Branch probes first require the
static site/path IDs above. Local workspaces may explicitly apply a plan or patch; GitHub temporary
clones should export it rather than pretending a mutation is durable.

The POC's selected branch detail includes a generated probe preview. It demonstrates the plan's
payload, but does not add a source-writing HTTP endpoint.

## Demo recordings

Both demos are synthetic, deterministic, safe, and tagged as demo data.

### Successful discounted order

`POST /orders` with a present customer, two order lines, and `WELCOME10`:

Timed rows are `handleCreateOrder -> placeOrder -> validateOrderRequest -> price -> assemble ->
save -> sendOrderConfirmation`. Pricing sub-steps and the created response are point events within
their owning spans rather than separate timed rows.

The timeline shows the validation loop count, the recognized-discount branch, safe monetary values,
and one opaque order token flowing through assembly, repository, and notification.

### Rejected order

`POST /orders` with no customer:

`handleCreateOrder -> placeOrder -> validateOrderRequest -> ValidationError -> catch ->
toErrorResponse -> 400`

Pricing, storage, and email spans are not observed in this recording. For the deterministic demo the
capture plan covers those callables, so the scenario is intentionally constructed not to invoke them;
the product UI still uses the conservative **not observed** label. The branch event identifies the safe
predicate (`customer.present=false`) that selected the error path.

## Acceptance criteria

- In **Telemetry** mode, a viewer launched without `--overlay` exposes **Off** plus **Synthetic demo**
  in Request data; outside that mode the telemetry widgets and runtime paint are absent.
  Selecting the source alone leaves telemetry unloaded; explicitly loading `demo` returns aggregate
  metrics and at least twelve request recordings.
- The Request trace sub-view uses observed milliseconds and renders the correct nested span order.
- Selecting success vs. rejection visibly changes both the executed rows and branch/data details.
- The same request selection visibly changes observed cards and proven execution wires on every graph
  map lens without changing node positions.
- Collapsed graph levels roll exact descendant evidence into their visible package/file/service card.
- Normal graph selection remains independent and visible while the request overlay is active.
- Unpainted code is described as not observed; graph-reference mismatch paints no runtime evidence.
- Every mapped row resolves to an existing `node.id`; unmapped rows remain visible.
- Branch pins show condition, outcome, safe attributes, source line, and a generated probe preview.
- Contract and pure timeline derivation have unit tests; CLI tests cover the mandatory env gate.
- Core, renderer, and CLI typechecks pass, followed by the full build and a visual smoke check.

## Run the POC

```bash
pnpm install
pnpm build
node packages/cli/dist/bin.js generate examples/orders-service \
  -o /tmp/orders-service.meridian.json
node packages/cli/dist/bin.js view /tmp/orders-service.meridian.json \
  --source-root examples/orders-service
```

In the browser:

1. Turn on **Telemetry** under **Overlays**, then in **Request data** choose **Synthetic demo**. Its
   only environment, **demo**, is visibly
   prefilled but remains unloaded until you click **Load**. From an empty Request trace or graph map,
   **Load synthetic demo** performs the same explicit action in one step.
2. On Map, switch between the `WELCOME10` and missing-customer requests in the request overlay.
3. Drill or switch Map / Service / UI and observe the same selection roll up to each visible graph.
4. Open Logic for `OrderRoutes.handleCreateOrder` and select **Request trace**.
5. Select a diamond branch pin to inspect its named input, outcome, source line, and generated probe.

`--overlay mock --env demo` remains available when a caller wants the demo source and environment
pre-filled at startup; the browser still requires the explicit Load action.

## Deferred

- Tempo/OTLP ingestion and trace search/pagination.
- Live recording control and tail sampling.
- Python or distributed multi-service instrumentation.
- Source mutation/apply/revert workflow.
- Full data lineage, production capture rollout, and clock-skew correction UI.
